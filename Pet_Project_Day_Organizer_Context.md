# Контекст пет-проекта: личный органайзер / планировщик дня

Дата фиксации: 25.06.2026

## 1. Назначение документа

Этот документ нужен как компактная база проекта. Его можно загрузить в другой чат, чтобы продолжить работу без потери контекста. Здесь собраны идея, правила работы, уровень пользователя, будущие функции и текущий план.

## 2. Роль главного чата

Этот чат считается главным чатом проекта. Здесь обсуждаются идея, общий план, приоритеты, архитектурные решения, состав функций и порядок разработки. Промты для других чатов даются только после обсуждения задачи, без спешки и без огромных ответов на много тем сразу.

Другие чаты могут использоваться отдельно для кода, архитектуры, тестирования, интерфейса или объяснений, но они не должны делать весь проект сразу. Их роль — помогать пользователю разрабатывать по шагам, объяснять, проверять и обучать.

## 3. Уровень пользователя

Пользователь примерно на уровне junior / стажёр:

- C++: базовый уровень / стажёрский уровень;
- Java: немного;
- Python: в основном умеет читать код и разбираться, может запускать с помощью подсказок;
- SQL: немного знает основы;
- Docker: немного запускал, в основном с помощью ИИ;
- тестирование: знает основы QA, чек-листы, проверки, баги, базовую логику тестирования.

Из-за этого проект нужно строить пошагово: маленькая задача, объяснение, запуск, проверка, исправление ошибок, потом следующий шаг. Не нужно сразу давать сложную архитектуру, большое количество файлов, паттерны уровня middle/senior или километровые промты.

## 4. Главная идея проекта

Проект — не просто программа для заметок. Основная идея сейчас: личный органайзер / планировщик дня с заметками, учётом, прогрессом и возможностью повторять удачные дни.

Главная ценность приложения: быстро составить план дня, легко менять его по ходу дня, отмечать выполненное и видеть прогресс. В будущем приложение может быть выложено публично и, возможно, монетизировано, но сейчас цель — учебный пет-проект с аккуратной основой.

Рабочая формулировка проекта:

> Локальный планировщик дня с заметками, учётом, прогрессом и возможностью копировать удачные дни.

## 5. Основные разделы приложения

### 5.1. План дня / Сегодня

Главная вкладка приложения. В ней пользователь видит текущий день и может быстро управлять задачами.

Что должно быть в будущем:

- список дел на день;
- быстрое добавление задачи;
- редактирование задачи;
- удаление задачи;
- отметка выполнения;
- статусы: не начато, в процессе, выполнено, отменено или перенесено;
- прогресс дня: например, 4 из 7 задач выполнено;
- возможность менять порядок задач;
- привязка ко времени, если нужно;
- перенос задачи на другой день;
- позже — напоминания.

### 5.2. Заметки

Раздел для обычных текстовых заметок. С него удобно начинать разработку, потому что на нём можно научиться хранить данные, делать CRUD и работать с SQLite.

Минимально:

- создать заметку;
- открыть заметку;
- отредактировать заметку;
- удалить заметку;
- сохранить после перезапуска приложения.

Позже:

- поиск;
- категории;
- теги;
- избранное;
- привязка заметки к дате или задаче;
- экспорт.

### 5.3. Учёт / Трекеры

Отдельная вкладка, не связанная напрямую с делами и напоминаниями. Это раздел для записей и отслеживания разных вещей.

Примеры:

- кто сколько должен;
- кому пользователь что дал;
- сколько чего съел;
- сколько воды выпил;
- сколько денег потратил;
- сколько времени ушло на учёбу или проект;
- привычки;
- личные счётчики.

Важно: этот раздел не должен смешиваться с задачами дня. Это именно учёт, журнал, трекеры и простые таблицы.

### 5.4. Будущие разделы

Позже можно добавить:

- календарь;
- статистику;
- шаблоны дней;
- настройки;
- темы оформления;
- экспорт и резервные копии;
- Android-версию.

## 6. Будущие сильные функции

### 6.1. Напоминания / будильник

Позже у задачи или заметки можно будет поставить напоминание:

- дата;
- время;
- повторение;
- активность;
- уведомление;
- звук или системное уведомление.

Эту функцию не делать в первой версии. Она должна быть в долгосрочном плане.

### 6.2. Копирование хорошего дня

Одна из ключевых будущих функций. Если день получился удачным, пользователь может скопировать его на другую дату и потом подправить.

Пример:

- был хороший план на понедельник;
- пользователь нажал «Скопировать день»;
- выбрал среду;
- все задачи/блоки дня перенеслись;
- пользователь отредактировал время и лишние пункты.

Позже из этого можно сделать шаблоны: «учебный день», «рабочий день», «выходной», «день проекта», «хороший день».

### 6.3. Прогресс и игровая механика

Игровая механика возможна, но лёгкая, без перегруза.

Идеи:

- процент выполнения дня;
- очки за выполненные задачи;
- серия дней;
- уровень;
- небольшие достижения;
- отметка «хороший день»;
- статистика за неделю.

Это добавлять только после того, как будет нормально работать план дня.

## 7. Стек и технический подход

### 7.1. Первая версия

Для первой desktop-версии лучше использовать:

- Python;
- Tkinter или CustomTkinter для интерфейса;
- SQLite для хранения данных.

Причина: это проще для старта, соответствует текущему уровню пользователя и позволяет быстро получить рабочую программу. Главное сейчас — понять логику приложения, хранение данных, CRUD, интерфейс и тестирование.

### 7.2. Android в будущем

Android пока не делать. Когда desktop-версия станет понятной, можно будет решить:

- переписать приложение на Flutter;
- сделать Android-версию отдельно;
- использовать Kotlin/Java;
- вынести данные в backend и делать разные клиенты.

Решение по Android нужно принимать позже, когда будет рабочее ядро.

### 7.3. Коммерческое будущее

В будущем проект можно попробовать выложить публично или монетизировать. Но сейчас не нужно делать оплату, аккаунты, сервер, облако и сложную аналитику.

Сейчас важно заложить:

- понятную структуру проекта;
- аккуратное хранение данных;
- нормальные названия файлов и функций;
- возможность расширения;
- приватность данных пользователя;
- резервное копирование в будущем;
- интерфейс, который реально удобно использовать.

## 9. Дорожная карта

### Этап 0. Уточнение идеи

Сейчас идёт именно этот этап. Обсуждаются функции, уровень пользователя и общий подход. Промты для других чатов пока не спешить делать.

### Этап 1. MVP заметок

Цель: научиться сохранять данные и сделать первую рабочую программу.

Функции:

- окно приложения;
- список заметок;
- создание заметки;
- редактирование;
- удаление;
- сохранение в SQLite.

### Этап 2. Минимальный план дня

Цель: сделать главную вкладку «Сегодня».

Функции:

- задачи на выбранную дату;
- добавление задачи;
- отметка выполнения;
- редактирование;
- удаление;
- прогресс дня.

### Этап 3. Учёт / трекеры

Цель: отдельная вкладка для записей и счётчиков.

Функции:

- создать тип учёта;
- добавить запись;
- посмотреть записи за день;
- простая таблица;
- базовая статистика.

### Этап 4. Улучшение удобства

Функции:

- поиск;
- фильтры;
- категории;
- быстрые действия;
- изменение порядка задач;
- перенос задачи на другой день.

### Этап 5. Копирование хорошего дня

Функции:

- скопировать задачи одного дня на другой;
- отредактировать копию;
- сохранить шаблон дня.

### Этап 6. Напоминания

Функции:

### Этап 7. Игровая механика и статистика

Функции:

- очки;
- серии дней;
- уровни;

### Этап 8. Подготовка к публичному продукту

Функции:

- резервные копии;

## 10. Правила работы над проектом

1. Не делать всё сразу.
2. Сначала обсуждать задачу, потом писать промт или код.
3. Ответы должны быть короткими и поэтапными.
4. Код давать маленькими частями.
5. После каждой функции пользователь сам запускает и тестирует.
6. Каждая новая функция проходит цикл: идея → реализация → запуск → проверка → исправление → запись результата.
7. Не усложнять архитектуру раньше времени.
8. Новые функции добавлять только после того, как текущая версия работает.
9. Учитывать будущую монетизацию, но не делать коммерческие функции в MVP.
10. После значимого обновления давать краткий блок «Для другого чата».

## 11. Как тестировать проект

Пользователь хочет учиться QA, поэтому тестирование должно идти параллельно разработке.

Для каждой функции желательно делать:

- чек-лист;
- позитивные проверки;
- негативные проверки;
- регрессионные проверки после исправлений;
- баг-репорт, если что-то не работает.

Пример для заметки:

- создать заметку с обычным заголовком;
- создать заметку с пустым заголовком;
- создать заметку с длинным текстом;
- изменить заметку;
- удалить заметку;
- закрыть и открыть приложение;
- проверить, что данные сохранились.

## 12. Что сейчас НЕ делаем

Пока не делать:

- Android;
- облачную синхронизацию;
- регистрацию;
- оплату;
- сервер;
- сложную игровую систему;
- сложную аналитику;
- автосинхронизацию;
- огромную архитектуру.

Сначала нужно получить маленькую, понятную и рабочую desktop-версию.

## 13. Текущий следующий шаг

Следующий логичный шаг — не писать код сразу, а коротко определить MVP 0.1:

- какие экраны будут в первой версии;
- что точно входит;
- что точно не входит;
- какие данные сохраняем;
- как пользователь будет проверять результат.

После этого можно будет подготовить первый промт для чата-разработчика или сразу начать делать структуру проекта.

## 14. Краткая выжимка для другого чата

Проект: личный органайзер / планировщик дня с заметками.

Главная цель: удобный план дня на каждый день, где можно быстро менять задачи, отмечать выполненное и видеть прогресс. Дополнительно нужны обычные заметки и отдельная вкладка «Учёт» для трекеров вроде «кто что должен», еды, воды, расходов, времени и личных счётчиков.

Будущие функции: напоминания/будильник, копирование удачного дня на другую дату, шаблоны дней, лёгкая игровая механика, статистика, Android-версия, возможная публикация и монетизация.

Текущий подход: начинать с маленькой desktop-версии. Предпочтительный стартовый стек: Python + Tkinter/CustomTkinter + SQLite. Пользователь уровня junior/стажёр: знает базово C++, немного Java, читает Python, немного SQL, основы QA и немного Docker. Нужно обучать по шагам, не делать весь проект сразу, давать короткие объяснения и обязательно просить пользователя запускать и тестировать.

Важно: не спешить с промтами. Сначала обсуждать идею и план. После значимых изменений давать блок «Для другого чата».
